# 离线 RAG 评测工具链详解

## 1. 这套工具链要解决什么问题

RAG 优化最容易出现的问题，是每次改完检索、切块、OCR、rerank 之后，只能靠手工问几条问题来“感觉”效果有没有变好。这种验证方式有几个明显缺点：

- 不可复现，今天测过的问题明天可能就忘了
- 很容易挑对自己有利的 case，导致评估失真
- 无法稳定比较不同版本的效果
- 文档一多以后，人工验证成本会迅速变高

所以这个项目里做了一套离线 RAG 评测工具链，目标不是替代线上真实用户反馈，而是提供一套可复现、低成本、能持续回归的本地评估流程。

这套工具链主要服务三个场景：

- 验证文档级召回是否比普通 chunk 拼接更完整
- 验证章节级问题时，检索结果是否能覆盖目标章节
- 在改动切块、检索、重排、OCR、metadata 加权之后，快速判断效果有没有退化

## 2. 工具链整体流程

整个流程可以拆成三步：

1. 先把原始资料复制到工作区，并归一化成统一编码的副本
2. 再从归一化后的文档中自动生成一批评测样本
3. 最后调用本地 `/api/rag/query` 接口跑这些样本，输出结果和汇总指标

对应脚本分别是：

- `test/rag-eval/normalize-study-resource.ps1`
- `test/rag-eval/generate-rag-eval-set.ps1`
- `test/rag-eval/run-rag-eval.ps1`

这三步的设计核心是：

- 不直接改原始资料
- 先做脏数据清洗，再生成样本
- 评测对象不是最终大模型回答，而是 RAG 检索上下文和命中片段本身

## 3. 第一步：资料归一化

### 3.1 为什么要先做归一化

实际资料经常来自不同来源，比如笔记、教程、爬下来的 Markdown、复制出来的文本、历史文档等。它们很容易存在这些问题：

- 编码不统一，可能有 UTF-8、UTF-16、GBK、GB18030、Big5
- 控制台显示正常，但程序解析会乱码
- 有些文件混杂乱码字符、替换字符、异常标点
- 原始文件不希望被脚本直接改写

如果不先做归一化，后面的样本生成和评测就会被脏数据污染，最终得到的结果没有参考价值。

### 3.2 归一化脚本做了什么

脚本：`test/rag-eval/normalize-study-resource.ps1`

输入：

- `SourceDir`：原始资料目录
- `WorkspaceDir`：工作区目录

执行后会创建三个核心产物：

- `workspace/copied`：原始目录的完整副本
- `workspace/normalized`：统一转成 UTF-8 的归一化文本副本
- `workspace/manifest.json`：每个文件的编码和可疑程度记录
- `workspace/summary.json`：整体统计信息

### 3.3 它具体怎么做编码处理

脚本会先注册 CodePages 编码支持，然后针对文本文件尝试多种解码方式。它会优先处理带 BOM 的文件，比如：

- UTF-8 BOM
- UTF-16 LE BOM
- UTF-16 BE BOM

如果没有 BOM，就按候选编码依次尝试：

- `utf-8`
- `utf-16-le`
- `utf-16-be`
- `gb18030`
- `gbk`
- `big5`

一旦找到看起来更合理的解码结果，就把文本统一转成 UTF-8 无 BOM，并做换行归一化。

### 3.4 它怎么判断文本是否可疑

归一化脚本里有一套简单但实用的文本质量检测逻辑，会统计这些信号：

- 替换字符 `�` 的数量
- 常见乱码模式数量
- 异常标点数量
- 拉丁字符和中文字符比例

如果这些指标超阈值，就会把文件标成 `suspicious`。这些信息会写到 `manifest.json` 里，供下一步生成样本时过滤脏数据。

### 3.5 为什么这一步很重要

这一步本质上是在做“评测数据治理”。

如果样本本身就来自乱码文件，那后续评测失败，到底是 RAG 没检索到，还是数据本身有问题，就分不清了。先把编码、文本质量、文件副本这些问题处理掉，后面的评测才有意义。

## 4. 第二步：自动生成评测集

### 4.1 为什么要自动生成评测集

人工写评测集当然更精确，但成本很高。我的项目当前阶段更重视的是：

- 能快速得到第一版评测集
- 能覆盖文件总结、按文件名总结、章节总结这些 RAG 高频场景
- 能随着资料目录变化快速重新生成

所以这里先采用了启发式自动生成方式，而不是完全人工标注。

### 4.2 生成脚本做了什么

脚本：`test/rag-eval/generate-rag-eval-set.ps1`

输入：

- `NormalizedDir`：归一化后的 Markdown 文档目录
- `OutputDir`：评测集输出目录
- `MaxDocuments`：最多处理多少文档
- `MaxSectionSamplesPerDocument`：每个文档最多生成多少个章节样本

输出：

- `rag-eval.jsonl`：逐条评测样本
- `rag-eval-summary.json`：样本统计

### 4.3 它筛选了哪些文档

当前脚本主要处理归一化目录下的 `.md` 文件，并且会跳过这些情况：

- 在 `manifest.json` 中已经标记为 `suspicious` 的文件
- 文件太短，内容不足以支撑评测
- 文件前部文本质量看起来依然很差
- 标题本身就是脏文本或乱码

这一步的目的，是尽量不要让坏数据进入样本集。

### 4.4 它生成了哪些样本类型

当前一共生成三类样本：

#### 1. `document_summary`

问题形式类似：

- 请总结《某某文档》这份文件的核心内容。

这类样本主要测试：

- 系统是否能命中文档整体上下文
- 文件级召回是否有效

#### 2. `file_name_summary`

问题形式类似：

- 根据文件名《某某文档》，总结这整个文件。

这类样本和第一类很接近，但更强调文件名信号是否能帮助检索到正确文档。

#### 3. `section_summary`

问题形式类似：

- 《某某文档》中“某章节”这一节主要讲了什么？

这类样本主要测试：

- 章节级检索是否命中正确 section
- 结构化文档切块和 metadata 加权是否起作用

### 4.5 它是怎么构造 expected answer 的

这套工具链当前不是用人工标注答案，而是用启发式方式构造“参考答案”：

- 文档级样本通常取第一段较长、较干净的段落作为 `expectedAnswer`
- 章节级样本则把目标章节内容裁剪压缩成 `expectedAnswer`
- 同时章节样本还会记录 `expectedSections`，用于后续判断检索上下文是否包含目标章节名

这意味着它不是一个严格的语义金标准，而是一套“第一版可运行回归集”。

### 4.6 为什么这种生成方式是合理的

因为当前目标不是做学术级评测，而是做工程回归验证。

只要样本能稳定覆盖你关心的几类能力，并且每次生成逻辑一致，就可以用来比较不同版本之间有没有退化。之后如果需要更高质量评测，再补人工标注集也可以。

## 5. 第三步：运行离线评测

### 5.1 评测脚本做了什么

脚本：`test/rag-eval/run-rag-eval.ps1`

输入：

- `BaseUrl`：本地后端地址
- `EvalFile`：评测集文件
- `OutputDir`：结果输出目录
- `BearerToken`：如果接口需要认证，就传 JWT
- `TopK`：查询时用的 topK
- `MaxSamples`：最多评多少条
- `TimeoutSeconds`：请求超时秒数

输出：

- `rag-eval-result-时间戳.jsonl`：逐条评测结果
- `rag-eval-summary-时间戳.json`：总体统计结果

### 5.2 它评测的是哪一层

这里评测的重点不是最终生成回答，而是 `/api/rag/query` 的检索结果，也就是：

- `response.data.context`
- `response.data.matches`

这么做的原因是，当前目标是验证 RAG 检索链路本身，而不是把大模型生成质量也混进来。否则你很难判断问题出在检索、重排、上下文组织，还是出在回答模型本身。

### 5.3 它怎么计算分数

脚本做了三类核心判断：

#### 1. `contextCoverage`

它会把 `expectedAnswer` 和返回的 `context` 做关键词覆盖率比较。

#### 2. `matchCoverage`

它会把 `expectedAnswer` 和所有命中的 `matches.content` 拼接结果做关键词覆盖率比较。

#### 3. `sectionHit`

如果样本有 `expectedSections`，它会检查章节名是否真的出现在检索上下文或命中片段里。

最后综合成一个简单分数：

- `score = contextCoverage * 0.7 + matchCoverage * 0.3 + sectionHit 加分`

并给出是否通过：

- 没有请求错误
- `contextCoverage >= 0.35` 或 `matchCoverage >= 0.45`
- 同时 `sectionHit` 为真

### 5.4 它为什么不用复杂 NLP 指标

因为当前场景更适合一个轻量、稳定、低依赖的本地评测方案。

如果上来就用 BLEU、ROUGE、BERTScore 或者再接一个 judge 模型，会带来更多不稳定性和额外依赖。现在这套方案虽然简单，但优点是：

- 容易落地
- 容易复现
- 适合工程回归
- 可以快速发现明显退化

## 6. 这套工具链的工程价值

### 6.1 它解决了什么工程问题

它解决的不是“让论文指标更好看”，而是工程上的这几个现实问题：

- 改了切块策略后，怎么知道文件总结能力是不是退化了
- 改了 metadata 加权后，章节命中能力有没有变好
- 改了 OCR 或编码处理后，脏数据是不是影响到了整体效果
- 每次改检索参数后，怎么低成本做回归

### 6.2 为什么我认为这套工具链值得写进项目亮点

因为它体现了一个很重要的工程意识：

- 不只是“做功能”
- 还要能“验证功能是否持续有效”

在 RAG 项目里，很多人能把上传、切块、embedding、查询跑通，但很少有人会补一套离线回归工具链。这个点很适合在面试里讲，因为它能体现你对系统可持续优化的理解。

## 7. 它和项目其他模块的关系

这套评测工具链和项目里的这些能力直接相关：

- 文档清洗
- OCR
- 结构化切块
- metadata 加权
- hybrid retrieval
- rerank
- document-aware 文件级召回

也就是说，它不是孤立存在的脚本，而是整个 RAG 优化闭环里的“验证层”。

完整闭环可以理解为：

1. 文档进入系统
2. 做抽取、OCR、切块、embedding
3. 查询时做检索和重排
4. 调整策略
5. 用离线评测重新验证
6. 再决定是否继续上线或保留该优化

## 8. 这套工具链的局限性

面试里如果被问到“你这套评测是不是已经很完整了”，不要说得太满。它现在仍然有明显边界：

- 参考答案是启发式生成，不是人工精标
- 目前更偏检索评测，不是端到端回答质量评测
- 主要覆盖文件总结和章节总结，不覆盖事实问答、跨文档推理、多跳问题
- 关键词覆盖率是轻量近似指标，不等于真正语义理解
- 章节命中判断依赖标题出现，不是更细粒度的语义对齐

这几个局限要主动承认，因为这反而会让你的表达更可信。

## 9. 如果继续做，我会怎么升级

后续如果继续扩展，我会优先做这些升级：

### 9.1 加人工标注小样本

先做一个数量不大但质量更高的人工标注集，作为高置信基准集。这样可以和启发式样本集互补。

### 9.2 增加更多任务类型

比如：

- 事实型问答
- 多段信息聚合
- 跨文档问题
- 含表格/图片 OCR 的问题
- 多轮追问场景

### 9.3 区分检索层和生成层评测

现在更偏检索层。后续可以拆成：

- 检索命中质量
- 最终回答质量
- 引用是否正确
- 幻觉率是否可控

### 9.4 引入更强的评测器

后续可以尝试：

- LLM-as-a-judge
- 更细的 semantic similarity 指标
- 对比不同 chunk、topK、rerank 配置下的 A/B 结果

## 10. 一句话总结

这套离线 RAG 评测工具链，本质上是我为 RAG 系统补上的一层“可复现验证能力”。它把原始资料归一化、样本生成、接口回归跑通，让 RAG 优化不再只靠感觉，而是能被持续比较和验证。
