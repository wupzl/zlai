# RAG 评测集标注、数据来源与效果结论

## 1. 先给最直接的答案

如果面试官问：

- 评测集是谁标注的？
- 数据哪里来的？
- 用什么工具评估？
- 优化效果怎么样？

最真实、最稳的回答应该是：

**当前这套评测集不是人工逐条精标的，而是我基于自己的真实资料库，通过归一化和启发式脚本自动生成的第一版工程回归集。评估工具是项目里自写的 PowerShell runner，直接调用 `/api/rag/query` 做接口级回归。它已经能支撑离线比较，但如果按严格标准说，目前还缺少一轮保留结果文件的真实在线评测，所以现在不能把优化效果说成“已经被完整量化验证过”。**

这个回答的关键是：

- 不吹成“人工高质量 benchmark”
- 也不把已有工作说得太轻
- 准确体现当前项目状态

## 2. 评测集是谁标注的

### 2.1 不是人工逐条标注

当前项目里的评测集，**不是人工一条一条写 question / answer / gold evidence 的精标集**。

从 `generate-rag-eval-set.ps1` 的逻辑可以明确看出来：

- question 是脚本按模板自动生成的
- expected answer 也是脚本从文档里抽出来的
- section 标题也是从 Markdown 结构自动提取的

所以严格来说：

**它是启发式自动生成样本，不是人工标注样本。**

### 2.2 更准确的说法

如果要更专业一点，可以说：

当前是“弱监督/启发式构造的第一版评测集”，主要用于工程回归，而不是论文式人工金标 benchmark。

### 2.3 为什么这样做

因为项目当前阶段更需要的是：

- 先快速建立一套可复现的回归机制
- 先覆盖最重要的业务场景
- 先让 RAG 优化能被比较

而不是一开始就投入大量成本做人工精标。

## 3. 具体是谁生成的

如果面试官继续追问“那到底是谁做的”，你可以这样答：

**是我自己设计生成规则并写脚本生成的，不是外包标注、也不是第三方公开 benchmark。**

这里要把“我自己写了工具链”和“不是我人工一条条标注”区分清楚。

也就是说：

- 规则是你设计的
- 脚本是你写的
- 样本是你生成的
- 但不是人工精标

## 4. 具体数据哪里来的

### 4.1 数据来源不是公开 benchmark

从当前脚本和目录来看，数据主要来自你自己的资料库副本，而不是像 HotpotQA、MS MARCO、NQ 这种公开 benchmark。

最典型的来源是：

- `D:\StudyResources\JavaStudyResource`

然后被复制进：

- `test/rag-eval/workspace-java-study-resource/copied`
- `test/rag-eval/workspace-java-study-resource/normalized`

### 4.2 当前仓库里已有两套样本来源

#### 1. `workspace-java-study-resource`

这是主样本来源，规模更大。

当前统计：

- 文档数：`150`
- 样本数：`492`
- 类型分布：
  - `document_summary = 148`
  - `file_name_summary = 148`
  - `section_summary = 196`

#### 2. `workspace-smoke`

这是小规模 smoke 数据集，用于快速验证。

当前统计：

- 文档数：`5`
- 样本数：`20`
- 类型分布：
  - `document_summary = 5`
  - `file_name_summary = 5`
  - `section_summary = 10`

### 4.3 为什么用这类数据

因为你当前项目重点优化的是：

- 文件级召回
- 文件名导向问答
- 章节级问答
- 结构化文档检索

而你自己的资料库里恰好有大量：

- Markdown 文档
- 标题结构明显的知识型资料
- 章节组织比较清晰的学习笔记/资料库

这类数据比随便找公开问答集更贴近你项目当前真实应用场景。

## 5. 评测工具用的是什么

### 5.1 不是第三方现成评测框架

当前评测工具不是 LangSmith、Ragas、TruLens 这种现成框架。

它用的是项目里自己写的 PowerShell 脚本：

- `normalize-study-resource.ps1`
- `generate-rag-eval-set.ps1`
- `run-rag-eval.ps1`

### 5.2 评测 runner 怎么工作

`run-rag-eval.ps1` 的逻辑是：

- 读取 `rag-eval.jsonl`
- 逐条取出 question
- 调用本地 `/api/rag/query`
- 拿到 `context` 和 `matches`
- 和样本里的 `expectedAnswer / expectedSections` 做覆盖率比较
- 输出逐条结果和 summary

### 5.3 评测重点是什么

当前评的是：

- 检索层
- 上下文组织层

而不是最终模型生成答案层。

这是一个非常重要的定位。

因为它意味着：

**当前工具链主要在回答“检索有没有命中、上下文组织得够不够”，而不是“最终大模型回答是不是最优”。**

## 6. 样本是怎么构造的

### 6.1 `document_summary`

生成方式：

- 从文档标题构造问题
- 从第一段较长、较干净的段落构造 `expectedAnswer`

### 6.2 `file_name_summary`

生成方式：

- 直接按文件名构造“根据文件名总结整个文件”这类问题
- `expectedAnswer` 依然通常来自第一段主要内容

### 6.3 `section_summary`

生成方式：

- 先解析 Markdown section
- 取 section 标题构造问题
- 从对应 section 内容裁剪出 `expectedAnswer`
- 把 section 标题记录到 `expectedSections`

### 6.4 所以这是什么类型的数据集

它本质上是：

**基于真实资料库、自动生成的结构化问答回归集。**

不是人工金标，也不是公开 benchmark。

## 7. 评估数据是怎么打分的

当前主要有三类指标：

### 7.1 `contextCoverage`

衡量：

- 返回的 `context`
- 对 `expectedAnswer` 的关键词覆盖率

### 7.2 `matchCoverage`

衡量：

- 返回的 `matches.content`
- 对 `expectedAnswer` 的关键词覆盖率

### 7.3 `sectionHit`

衡量：

- 目标章节标题有没有出现在 `context` 或 `matches` 中

### 7.4 最终通过逻辑

当前通过条件大致是：

- 没有请求错误
- `contextCoverage >= 0.35` 或 `matchCoverage >= 0.45`
- `sectionHit = true`

这说明当前评测更偏：

- 检索正确性
- 上下文覆盖度
- 章节命中能力

## 8. 优化效果怎么样

这里一定要实话实说。

### 8.1 现在能说的

你现在能说的是：

- 我已经搭好了离线评测工具链
- 我已经生成了两套样本集
- 这套评测集确实是围绕文件总结、文件名总结、章节问答这些目标场景构建的
- 它已经可以用来比较不同版本的检索和上下文组织表现

### 8.2 现在不能硬说的

但你现在**不能硬说**：

- 我已经通过真实在线评测证明优化效果提升了多少
- 我已经有一套稳定的 pass rate 结果
- 我已经能量化说明 hybrid retrieval / rerank / document-aware 带来了多少提升

原因很简单：

当前仓库里虽然有样本集产物，但没有看到本次环境重新跑出的：

- `rag-eval-result-*.jsonl`
- `rag-eval-summary-*.json`

而且当前机器上：

- 后端没启动
- `127.0.0.1:8080` 不可达
- Docker daemon 也没启动

所以现在最真实的结论是：

**工具链和样本集已经具备，但优化效果还没有在当前环境下通过真实在线回归结果被重新量化确认。**

## 9. 面试里应该怎么回答“效果怎么样”

最稳的说法是：

我当前已经完成了离线评测工具链和第一版样本集，重点覆盖文件级总结和章节级问答场景。它已经能支撑我做工程回归，但如果按严格标准说，当前还需要补一轮在目标环境上重新执行并保留结果文件的在线评测，才能把优化效果量化得更扎实。所以我现在更愿意说“已经具备验证能力”，而不是夸成“已经完整证明提升幅度”。

这个回答虽然保守，但很专业。

## 10. 如果面试官问“那你为什么还敢写这个点”

你可以这样答：

因为这个点的价值不只是结果数字，而是我已经把“如何验证 RAG 优化”这条链路搭起来了。很多项目只做到功能层，没有做到验证层。我现在已经把资料归一化、样本生成、接口回归和评分逻辑打通了，下一步就是把它跑成真正可持续使用的回归体系。

这个回答的重点是：

- 不是说自己已经全部做完
- 而是说明自己已经把正确的工程方向搭起来了

## 11. 你最应该背熟的几句话

- 当前评测集不是人工精标，而是我基于真实资料库启发式生成的工程回归集
- 数据主要来自我自己的学习资料库副本，不是公开 benchmark
- 评测工具是项目里自写的 PowerShell runner，不是第三方评测平台
- 当前主要评的是 `/api/rag/query` 的检索层和上下文组织层，不是最终回答层
- 现在已经有样本集，但还缺一轮当前环境下重新执行并保留结果文件的真实在线评测

## 12. 一句话总结

当前这套 RAG 评测数据，**是你自己基于真实资料库生成的启发式工程回归集，而不是人工金标 benchmark；评测工具也是自己写的 runner；验证能力已经搭起来了，但优化效果还没有在当前环境下通过真实在线结果被完整量化证明。**
