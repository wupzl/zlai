# 离线 RAG 评测相关面试点

## 1. 面试官为什么会对这个点感兴趣

因为它能体现的不是“你会不会调 prompt”，而是：

- 你有没有评估意识
- 你会不会做工程闭环
- 你是否知道 RAG 优化不能只靠手感
- 你是否能区分检索问题和生成问题

如果你能把这个点讲清楚，面试官通常会觉得你不只是把系统跑起来了，而是真的考虑过如何持续优化。

## 2. 最核心的标准回答

### Q1：你为什么要做离线 RAG 评测工具链？

建议回答：

因为 RAG 优化如果只靠手工问几条问题，很难复现，也很难比较不同版本之间的差异。我做这套工具链的目的，是把 RAG 优化变成一个可回归、可比较、低成本验证的流程。这样每次改切块、检索、rerank、OCR 或 metadata 加权之后，都能快速检查效果有没有退化。

### Q2：这套工具链整体流程是什么？

建议回答：

整体分三步：
- 第一步把原始资料复制到工作区并做编码归一化，避免直接污染原始资料，也减少乱码文件对后续评测的影响
- 第二步从归一化后的 Markdown 文档中自动生成评测样本，当前主要覆盖文件总结、按文件名总结、章节总结三类问题
- 第三步用这些样本去调用本地 `/api/rag/query` 接口，比较返回的 context 和 matches 对 expected answer 的覆盖情况，并输出汇总结果

### Q3：为什么你不直接人工写完整评测集？

建议回答：

人工标注当然更准，但成本高。当前阶段我先做的是一套启发式、低成本、可快速迭代的第一版工具链。它的目标不是学术评测，而是工程回归。后续如果继续升级，我会再补高质量人工标注集作为基准。

## 3. 脚本细节相关面试点

### Q4：归一化脚本具体解决了什么问题？

建议回答：

主要解决三个问题：
- 原始资料编码不统一，比如 UTF-8、UTF-16、GBK、GB18030、Big5 混在一起
- 有些文件存在乱码、异常标点或者替换字符
- 不希望脚本直接修改原始资料目录

所以脚本会先复制一份到工作区，再尝试多种编码解码，把文本统一转成 UTF-8，同时输出 manifest，记录每个文件的编码和是否可疑。

### Q5：为什么要做 `manifest.json`？

建议回答：

因为评测前需要先把坏数据识别出来。`manifest.json` 里会记录每个文件的编码、文本长度、是否可疑、可疑原因等信息。下一步生成样本时，可以直接跳过这些高风险文件，避免脏数据污染评测集。

### Q6：评测集目前有哪些类型？为什么选这几类？

建议回答：

当前有三类：
- `document_summary`
- `file_name_summary`
- `section_summary`

因为这几类问题最贴近我项目里重点优化的能力，也就是 document-aware 文件级召回和章节级问答。先把这几类高价值场景跑通，比一开始就铺很大的题型更有效。

### Q7：你是怎么构造 expected answer 的？

建议回答：

现在主要用启发式方式构造。比如文档总结类样本，会从文件里选第一段比较长、比较干净的段落作为 expected answer；章节类样本，会把对应 section 的内容裁剪成 expected answer。同时章节样本还会记录 expected section 标题，便于后面判断有没有命中目标章节。

## 4. 评测指标相关面试点

### Q8：你怎么判断一条样本是否通过？

建议回答：

当前是轻量规则：
- 先比较 expected answer 和返回 context 的关键词覆盖率
- 再比较 expected answer 和所有 matches 拼接结果的关键词覆盖率
- 如果样本带有章节信息，再检查目标章节标题是否出现在 context 或 matches 里

只要没有请求错误，并且 coverage 达到阈值，同时章节命中条件满足，就认为通过。

### Q9：为什么你评的是 `/api/rag/query`，不是最终模型回答？

建议回答：

因为我当前更关注 RAG 检索链路本身。最终回答会同时受到检索和大模型生成的影响，如果直接评最终答案，就很难区分问题是出在检索、上下文组织，还是出在模型生成。所以我先把检索层单独拉出来评，这样定位问题更清楚。

### Q10：为什么不用 BLEU、ROUGE、BERTScore 或 LLM judge？

建议回答：

当前阶段我更看重的是轻量、稳定、低依赖和可复现。BLEU、ROUGE 这类指标对中文和摘要类问题不一定稳定，LLM judge 虽然更强，但会引入额外成本和不确定性。现阶段这套关键词 coverage + section 命中的方案虽然简单，但更适合做工程回归。

## 5. 设计取舍相关面试点

### Q11：这套工具链最大的价值是什么？

建议回答：

最大的价值是把 RAG 优化从“凭感觉”变成“可回归验证”。它不能保证系统一定很好，但能保证你每次改动之后，知道自己有没有把某些能力做坏。

### Q12：这套工具链和你的 RAG 优化有什么关系？

建议回答：

它是验证层。比如我做了结构化切块、metadata 加权、document-aware 文件级召回、rerank 这些优化之后，就可以用这套工具链去验证：文件总结能力有没有提升，章节命中能力有没有更稳定。它不是孤立脚本，而是整个 RAG 优化闭环的一部分。

### Q13：你这套工具链的局限是什么？

建议回答：

局限主要有：
- 参考答案是启发式生成，不是人工精标
- 现在更偏检索层评测，不是端到端问答评测
- 样本类型还不够丰富，暂时主要覆盖文件总结和章节总结
- coverage 指标比较轻，不等于完整语义理解

这几个局限我会主动承认，因为它现在本来就是一个工程回归工具，不是论文级 benchmark。

## 6. 深挖追问与回答

### Q14：如果评测通过率下降了，你怎么排查？

建议回答：

我会按链路拆：
- 先看是不是样本数据变脏了，比如归一化或样本生成逻辑有变化
- 再看 retrieval 参数，比如 topK、阈值、metadata 权重、rerank 是否改过
- 再看 chunk 策略是不是导致文档结构被打散
- 如果检索 context 没问题，再看是不是 query 改写或上下文组织出了问题

核心思路是先分层定位，不会一上来就怀疑大模型。

### Q15：如果让你继续升级这套工具链，你会怎么做？

建议回答：

我会优先做四件事：
- 加一小批人工精标集，作为高置信评测基准
- 扩充样本类型，比如事实问答、跨文档问题、多跳问题
- 把检索层评测和最终回答评测分开
- 尝试引入更强的评估器，比如 LLM judge 或语义相似度指标

### Q16：为什么这个点适合写进简历？

建议回答：

因为它能体现我不是只会做功能，还会考虑怎么验证功能、怎么支持持续优化。很多 RAG 项目只写上传、切块、embedding、检索，但没有补验证链路。这个点更能体现工程能力。

## 7. 你最好背熟的 6 句话

- 我做离线 RAG 评测，不是为了学术指标，而是为了工程回归
- 这套工具链先做数据归一化，再生成样本，最后调用 `/api/rag/query` 跑回归
- 当前重点评的是检索层，而不是最终生成层
- 样本主要覆盖文件总结、按文件名总结、章节总结三类问题
- 指标虽然轻量，但优点是稳定、低依赖、容易复现
- 它最大的价值，是把 RAG 优化从凭感觉变成可比较、可验证的流程

## 8. 面试时的表达提醒

讲这个点时要避免两种错误：

### 错误 1：说得太学术

不要把它包装成“完整 benchmark 平台”。它现在更准确的定位是“离线回归工具链”。

### 错误 2：说得太轻

也不要说成“我写了几个脚本顺手测一下”。它本质上是你给 RAG 系统补上的验证层，体现的是工程方法论。

最稳的说法是：

我做了一套面向工程回归的离线 RAG 评测工具链，用来验证文件级召回和章节级问答这两类能力在每次优化后有没有退化。它不是论文 benchmark，但很适合支撑项目持续迭代。
